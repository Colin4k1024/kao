基于 React + Rust 的类若依(RuoYi)企业级后台管理系统需求规格说明书宏观架构定位与技术栈基建需求本项目旨在构建一个具备高度可伸缩性、安全内生性以及可长期维护性的后台管理系统，其核心业务逻辑与权限模型全面对标业界经典的若依（RuoYi）系统，但在底层架构上全面拥抱现代化的全栈技术栈。系统采用 Rust（依托 Axum Web 框架与 Tokio 异步运行时）作为高并发、内存安全的后端底座，前端采用现代化的 React 生态系统（结合 Vite 构建工具与 TypeScript 强类型约束）进行彻底的前后端分离重构。系统的根本目的在于提供一套开箱即用的“数字底座”，将业务线开发人员从底层架构搭建与权限逻辑编写中解放出来。为确保系统的高效运行与高可维护性，底层技术选型基准如下：后端技术栈：使用 Rust 语言，Web 框架选用生态成熟且人体工程学优秀的 Axum。关系型数据库交互采用 SQLx（提供编译期安全的 SQL 校验）或 SeaORM 等现代 Rust ORM 框架。前端技术栈：基于 React 19，使用 Vite 作为极速构建工具。UI 呈现采用 utility-first 的 Tailwind CSS 结合 Shadcn UI 组件库构建高质量、可访问的交互界面。状态与数据流管理：依靠 TanStack Query (React Query) 处理复杂的服务端状态同步与缓存，局部表单状态管理采用 React Hook Form 与 Zod 数据校验库。全局业务模块与组织架构基建需求矩阵系统在业务逻辑层面需内置极其丰富的基础功能模块，这些模块需经过深度领域抽象，提供通用的企业组织架构管理模型与权限管控机制。核心权限管控与组织架构图谱系统需构建一套极其严密且符合国际标准的 RBAC（基于角色的访问控制）模型体系。该体系通过多维度的实体数据表关联，实现灵活且细粒度的人员、机构与权限矩阵的动态绑定。模块核心命名业务规则映射与生命周期功能定义用户管理 (User Management)作为整个后台系统的核心操作主体数据中枢。不仅承载着系统登录账户生命周期的全量维护（包括创建、分配、封禁与信息修正），更是后续所有角色指派、权限拦截以及底层审计日志溯源的绝对根节点实体 。部门管理 (Department Management)用于精确配置和数字化定义企业内部复杂的层级与组织机构（如总公司、分中心、事业部、业务小组等）。系统在底层采用拓扑树状结构（Tree Structure）展现从属关系，这一数学集合基础是后续实现“基于机构的数据权限向下穿透过滤”的根本物理依托 。岗位管理 (Post Management)独立于角色的另一种身份标识，用于明确标记和配置系统用户在真实组织结构内所担任的具体行政或业务职务。这为基于行政职级的审批流节点流转提供了数据支撑 。角色管理 (Role Management)权限聚合与下发的核心枢纽。不仅负责将零散的系统菜单与按钮操作权限进行打包绑定，更支持动态设置该角色针对机构层级的数据范围划分（例如限制某销售总监角色仅能跨部门阅览特定区域的数据集） 。菜单管理 (Menu Management)系统前端视图展现与后端功能调用的双向映射字典。负责全盘配置系统的左侧导航菜单结构、操作权限的字符定义以及前端界面中各类具有破坏性操作按钮的显隐鉴权标识 。动态系统配置与自动化运维枢纽为最大限度地降低系统在运行期对硬编码的依赖比例，提高平台交付后的动态调整能力，需实现以下动态配置模块：模块核心命名业务规则映射与生命周期功能定义字典管理 (Dictionary Management)针对系统中高频出现、具有强业务分类属性且相对固定的枚举数据（如性别、订单状态、资产类型等）进行统一的集中化维护。通过数据字典机制，实现了前后端关于状态码的彻底解耦与动态渲染映射 。参数管理 (Parameter Configuration)提供全局变量的持久化存储能力。支持将系统运行期的关键环境变量、高频业务开关参数提取至数据库中进行动态在线热配置，彻底规避了因微小参数修正而导致的全系统停机重启与重新编译部署风险 。通知公告 (Notice Broadcasting)建立系统内部横向到边、纵向到底的信息广播机制。支持富文本图文并茂的系统级全局公告发布与日常维护，满足企业日常行政指令管理与重要系统运维事件的强制通达需求 。定时任务 (Background Jobs)深度集成 Rust 原生的异步后台任务调度队列（如 background-jobs 或类似的持久化任务流转引擎），替代传统 Java 体系的 Quartz。支持运维人员通过直观界面配置调度表达式，动态添加、挂起或删除异步批处理任务，并提供详尽的执行日志溯源。研发效能提升与全自动辅助工具链系统需内置高度成熟的自动化代码生成器模块。该引擎支持直接连通底层数据库解析表结构元数据，一键逆向生成涵盖后端 Rust 逻辑代码（包括 Axum 路由层、业务逻辑 Service 层、基于 SeaORM/SQLx 的持久层实现代码）以及前端 React 视图组件的完整 CRUD 生命周期代码。同时，通过 Rust 宏与 OpenAPI 规范的结合，实时动态生成 Swagger/Redoc 接口文档。全方位数据访问隔离与后端权限防线需求系统的核心壁垒在于其无死角的权限管控引擎，需通过前后端协同与中间件技术构建三级防御装甲矩阵。基于中间件与 AST 解析的底层数据范围隔离 (Data Scope)在 Rust 后端架构中，解决复杂的“基于机构的数据权限向下穿透过滤”是一项工程挑战。本系统要求弃用传统面向切面编程（AOP），转而采用 Rust 生态下的中间件（Middleware）结合数据库层拦截。通过定义基于 Axum 的提取器（Extractors）来获取当前上下文的用户会话与角色权限标识，并利用 SeaORM 强大的 AST（抽象语法树）解析与修改能力，或者集成成熟的 Casbin-RS 授权引擎，在真正执行 SQL 查询前，动态将权限过滤的 WHERE 条件组装至原始查询语句中。系统需内建五种颗粒度极其明确的数据权限常量标量：数据作用域枚举常量 (Data Scope Constant)底层 SQL 动态组装行为与业务隔离含义解析1 - 全部数据权限 (DATA_SCOPE_ALL)赋予操作者最高级别的数据物理阅览权。在此模式下，拦截器将主动休眠，忽略任何 WHERE 子句的注入限制，使得该超级用户得以无阻碍地拉取全表全量业务数据 。2 - 自定数据权限 (DATA_SCOPE_CUSTOM)赋予系统管理员极高的跨部门配置自由度。底层框架通过复杂的表关联查询逻辑，只提取该角色在后台配置页面被系统管理员人为勾选、强关联的个别目标部门的数据集合 。3 - 部门数据权限 (DATA_SCOPE_DEPT)采用极其严格的本部门匹配约束红线。SQL 语法树拼接逻辑将强制限制检索出的结果集中，其业务数据的归属机构字段（dept_id）必须与当前操作用户的部门 ID 保持绝对数学相等 。4 - 部门及以下数据权限 (DATA_SCOPE_DEPT_AND_CHILD)完美支持针对组织架构拓扑树的自顶向下穿透查询。系统不仅允许检索本部门数据，还向下兼容读取所有嵌套子部门产生的数据，契合企业高管的全局审计需求 。5 - 仅本人数据权限 (DATA_SCOPE_SELF)实施最严苛的物理级数据隔离级别。在此常量作用下，SQL 拼接引擎将彻底忽略部门层面的关联，直接校验业务数据表中的核心用户标识字段（如 user_id）是否等同于当前执行请求的登录用户 ID 。视图层路由控制与后端 API 鉴权联动在前端 React 应用中，左侧导航与页面路由须基于用户权限动态生成。前端路由引擎需采用 TanStack Router，替代传统的静态路由方案。对于按钮级别的微观操作控制，系统需开发专属的 React 自定义 Hooks（如 usePermissions）或包装组件（如 <AuthGuard required="system:user:add">），通过读取上下文中存储的 JWT 权限声明（Claims），在虚拟 DOM 层面对无权按钮组件进行物理隐藏或销毁。在后端 Axum 服务层，需利用鉴权中间件系统（如基于 Tower-HTTP 实现的身份验证层），拦截任何未携带合法 JWT 凭据或试图越权调用的 HTTP 请求。标准化前端交互与组件封装体系为保证跨业务线的操作一致性并适应多终端自适应布局，前端系统须基于现代 React 生态进行组件化封装。非阻塞式弹窗流转与数据交互引擎系统全面采用基于模态框（Dialog/Modal）的非阻塞交互。底层需依托 Shadcn UI 与 Radix UI 的无障碍（Accessibility）组件集，封装全局通用的 useDialog Hook，接管弹窗的渲染、关闭回调及自适应宽度。针对复杂的 CRUD 表格业务流转，彻底弃用传统 BootstrapTable，转而引入强大的 TanStack Table 引擎作为核心的无头（Headless）数据网格解决方案。该方案需支持服务端分页、多列排序与深度筛选同步。同时，将高频的异步数据查询、提交与删除操作封装进定制化的 React Query Hooks（例如 useAddRecord, useDeleteRecords）中，不仅实现优雅的网络加载状态（Loading State）展示与按钮防抖禁用，更能直接接管接口响应成功后的本地缓存失效与表格数据重绘。表单数据录入与生命周期纳管将由 TanStack React Form 或 React Hook Form 接手，结合强大的 Zod 强类型 Schema 定义器，实现极致的前后端验证逻辑与数据模型统一。系统底层工程结构与领域物理切分规范为了在日常开发维护期向开发团队展示极致的软件工程学美感，系统无论是在后端 Rust 代码还是前端 React 工程中，都要求实施严格的领域切割。Rust 后端 Cargo Workspace 模块切割为杜绝“大泥球”架构崩塌危机，整个 Rust 后端项目采用 Cargo Workspace 的分包管理机制与模块化（Modular）架构设计：src/features/ 或独立的子 Crate 用于管理业务域模块。每一项独立业务（如 user 用户管理）其物理结构必须严格包含：model.rs：承载领域数据结构定义与核心验证逻辑。repo.rs：封装基于 SQLx 或 SeaORM 的底层数据库持久化操作引擎。service.rs：编排纯粹的业务逻辑与权限计算流转。routes.rs：统筹管理该模块向外暴露的 Axum HTTP 处理器与路由网关接口。src/common/ 模块：集中存放与业务无关的通用组件库，例如自定义的 Axum 错误转换器、JWT 令牌解析中间件、以及统一格式化的 JSON 响应体结构定义。React 前端特征驱动结构 (Feature-based Organization)前端代码库同样摒弃传统的扁平文件堆叠，采用按业务特征划分的纵深路由架构：src/
├── routes/              # TanStack Router 页面路由拓扑网
│   ├── dashboard/       # Dashboard 业务域控制群
│   │   ├── index.tsx    # 数据概览大盘页面
│   │   ├── users.tsx    # 用户管理主干页面
│   │   └── settings.tsx # 权限与参数配置控制台
├── components/          # 跨业务线共享的通用原子组件池
│   └── ui/              # Shadcn UI 底层渲染组件集合
└── api/                 # 基于 Axios / fetch 的底层网络通信封装层
系统级监控预警与全方位可观测性基建虽然我们放弃了 Java 生态下的 Spring Boot Admin，但对于底层服务器的健康度与应用级指标监控，Rust 提供了更为硬核的原生探针。后端系统需深度集成跨平台系统监控基建（如 sysinfo Crate），它能以极低的性能损耗直接穿透至操作系统内核层，实时采集当前宿主机或容器环境下的 CPU 多核心负载率、物理与虚拟内存分配快照、甚至是各个挂载磁盘的 I/O 读写压力指标。通过 Axum 路由暴露专门的 /metrics 或 /system-info 端点探针，可源源不断地向前端管理大屏或其他日志收集哨兵（如 Prometheus）输送第一手应用指标时间序列流。此外，借助 Rust 生态的 tracing 宏与日志门面，系统亦能实现类似动态控制日志级别排障的生产级高可用防御特性。
